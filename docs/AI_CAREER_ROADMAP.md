## 3. Database Schema

The database utilizes three main PostgreSQL tables created with SQLAlchemy and managed via Alembic:

### 3.1 `career_roadmaps`
This table represents the root entity for a career roadmap session.
* `id` (UUID): Primary key, autogenerated random UUID.
* `user_id` (UUID): Foreign key to `users.id` (on delete CASCADE).
* `resume_id` (UUID, Nullable): Foreign key to `resumes.id` (on delete SET NULL).
* `target_role` (VARCHAR(100)): The role the candidate is aspiring to.
* `current_role` (VARCHAR(100), Nullable): The candidate's current role.
* `experience_level` (VARCHAR(50)): ENTRY, MID, SENIOR, LEAD, or EXECUTIVE.
* `target_industry` (VARCHAR(100), Nullable): The target business sector.
* `roadmap_status` (VARCHAR(50)): Status tracking (PENDING, IN_PROGRESS, COMPLETED, FAILED).
* `estimated_duration_months` (INTEGER): Defaulting to `ROADMAP_DEFAULT_MONTHS` or `12`.
* `current_readiness_score` (INTEGER): Estimated readiness percentile (default 50).
* `provider` (VARCHAR(50), Nullable): The AI platform provider used.
* `model` (VARCHAR(100), Nullable): The AI LLM model name used.
* `prompt_version` (VARCHAR(20), Nullable): Prompt version.
* `metadata` (JSONB, Nullable): Dynamic metadata payload mapping to SQLAlchemy column.
* `created_at` (TIMESTAMP): Creation date.
* `updated_at` (TIMESTAMP): Updating date.

### 3.2 `roadmap_milestones`
This table defines phases or steps required to achieve the career transition goal.
* `id` (UUID): Primary key.
* `roadmap_id` (UUID): Foreign key to `career_roadmaps.id` (on delete CASCADE).
* `phase_number` (INTEGER): Phase order identifier.
* `title` (VARCHAR(100)): Title of the milestone.
* `description` (TEXT, Nullable): Detailed step instructions.
* `duration` (VARCHAR(50), Nullable): Estimated duration (e.g. "2 months").
* `order_index` (INTEGER): Milestone sequencing order.
* `status` (VARCHAR(50)): NOT_STARTED, IN_PROGRESS, COMPLETED.

### 3.3 `learning_recommendations`
This table stores resources recommended to cover the skill gaps.
* `id` (UUID): Primary key.
* `roadmap_id` (UUID): Foreign key to `career_roadmaps.id` (on delete CASCADE).
* `category` (VARCHAR(50)): Type of recommendation (e.g. Skill, Project, Course, Certification).
* `title` (VARCHAR(100)): Title of recommendation.
* `description` (TEXT, Nullable): Context or details.
* `priority` (VARCHAR(50)): LOW, MEDIUM, or HIGH.
* `resource_url` (VARCHAR(500), Nullable): Reference link.
* `estimated_hours` (INTEGER, Nullable): Estimated time.

## 4. API Specification & Examples

All routes are prefix-mounted under `/api/v1/ai/roadmap` and require a valid session authorization token.

### 4.1 Endpoints

1. **POST `/generate`**
   * **Summary**: Generate AI Career Roadmap.
   * **Description**: Receives request payload, checks database cache for an identical completed roadmap (same target/current role, experience level, target industry, resume, duration) to avoid duplicate LLM invocations. If cached, returns it; otherwise, creates a pending shell, triggers the generation engine, and returns the fully populated roadmap.
   * **Input**: `RoadmapCreate` JSON body.
   * **Returns**: `201 Created` with `RoadmapResponse`.
   * **Example Payload**:
     ```json
     {
       "resume_id": "8f3c760b-8531-4871-b0db-b0cb1358ef21",
       "target_role": "AI Architect",
       "current_role": "Backend Engineer",
       "experience_level": "SENIOR",
       "target_industry": "Tech",
       "estimated_duration_months": 12
     }
     ```

2. **POST `/{id}/generate`**
   * **Summary**: Generate or Regenerate Roadmap for existing ID.
   * **Description**: Forces generation/regeneration of the career roadmap details for a specific existing roadmap ID, bypassing the cache check.
   * **Returns**: `200 OK` with `RoadmapResponse`.

3. **GET `/history`**
   * **Summary**: List user's career roadmaps history.
   * **Returns**: `200 OK` with `RoadmapHistory` containing roadmaps list.

4. **GET `/{id}`**
   * **Summary**: Get detailed career roadmap.
   * **Description**: Returns detailed milestones and learning recommendations for the specific roadmap (verifying user ownership).
   * **Returns**: `200 OK` with `RoadmapResponse`.

5. **DELETE `/{id}`**
   * **Summary**: Delete a career roadmap.
   * **Returns**: `200 OK` success dictionary.

## 5. Prompt Design & Registry Integration

The system registers prompts under the `career_roadmap` category in the global `PromptRegistry`. All prompts use YAML front-matter, support semantic versioning, and strictly output JSON format.

### 5.1 Prompt Templates:
* `system_prompt.jinja`: Directs LLM to act as the Career Architect, enforce strict JSON structure, and output minified JSON only (no markdown quotes or conversational text).
* `career_roadmap.jinja`: Resolves core roadmap details and coordinates phases.
* `learning_path.jinja`, `projects.jinja`, `certifications.jinja`, `timeline.jinja`, `skill_gap.jinja`: Modular prompt layouts reserved for focused recommendations.

## 6. Generation Flow

The following sequence details how the engine operates:
1. **API Call**: User posts to `/generate`.
2. **Cache Check**: The engine checks the DB for an existing completed roadmap matching the exact input payload. If found, returns it immediately.
3. **Shell Initialization**: A database record is created in `PENDING` status.
4. **Context Building**: `RoadmapContext` aggregates all user metadata (Resume parsing, ATS Score, Cover Letters, GitHub analytics, Interview sessions).
5. **Prompt Resolving & Variable Filtering**: Prompt variables are prepared and filtered to match the Jinja AST's expected variables before validation, avoiding strict unknown-field failures.
6. **LLM Invocation**: Calls `AIService.execute` with `parser_type="markdown_json"`.
7. **JSON Validation & Retry**: The JSON structure is validated using Pydantic `AICareerRoadmapResponse`. If validation fails or JSON is malformed, it retries exactly once.
8. **DB Persistence**: Saves milestones (`RoadmapMilestone`) and recommendations (`LearningRecommendation` mapping skills, projects, certifications, resources).
9. **Status Update**: Updates the root roadmap record to `COMPLETED` and saves latency and generation metadata inside `roadmap_metadata`.

## 9. Skill Gap & Learning Recommendation Engine (Part 3)

The **Skill Gap & Learning Recommendation Engine** analyzes a user's background context against a target role to identify specific professional gaps and construct structured learning roadmaps.

### 9.1 Gaps Analyzed
The engine compares the candidate profile context (including resume, parsed skills, ATS score, job match records, GitHub repository insights, and interview history) to identify gaps across seven categories:
1. **Technical Gaps**: Essential hard/programming skills (e.g. system design, data structures, algorithms).
2. **Soft Skill Gaps**: Professional/relational skills (e.g. leadership, collaboration, agility).
3. **Domain Knowledge Gaps**: Industry-specific expertise (e.g. FinTech regulations, healthcare compliance).
4. **Tool Gaps**: Command-line utilities, DevOps instruments (e.g. Git, Docker, Kubernetes, CI/CD).
5. **Framework Gaps**: Framework expertise (e.g. FastAPI, React, Spring Boot).
6. **Communication Gaps**: Presentational and stakeholder alignment (e.g. stakeholder management).
7. **Confidence Gaps**: Mindset hurdles (e.g. imposter syndrome, public speaking).

Each identified gap is ranked by severity: `High`, `Medium`, or `Low`.

### 9.2 Personalized Learning Plan Structure
Recommendations are generated to align with target role, experience level, preferred industry, and current readiness score. The generated learning plan consists of:
- **Weekly Plan**: Week-by-week syllabus outlining topics, objectives, focus areas, and daily tasks.
- **Monthly Goals**: Milestones to hit every 30 days.
- **Quarterly Goals**: Long-term goals.
- **Practice Schedule**: Recommended coding/practice rhythm.
- **Certification Suggestions**: Target credentials (e.g. AWS Certified Developer).
- **Books & Courses**: Curated educational materials.
- **Hands-on Projects & Open Source**: Practical implementation milestones.
- **Interview Practice**: Target mock interview drills.

## 13. Career Timeline & Milestone Planner

Phase 11 Part 4 implements the Career Timeline & Milestone Planner to convert a generated career roadmap into a detailed, structured execution timeline.

### 13.1 Timeline Architecture

The timeline planner operates on top of the completed career roadmap structure:
1. **Inputs**: Takes a reference `roadmap_id` and optional parameters.
2. **Context Compilation**: Gathers the original roadmap's phases/milestones to build prompt variables.
3. **Template Registry**: Uses `timeline.jinja`, `milestones.jinja`, `monthly_plan.jinja`, and `weekly_schedule.jinja` dynamically registered.
4. **Execution & Parser**: Executes LLM calls using the global `AIService`, applying strict schema validations and retrying once on failures.
5. **Persistence**: Saves the output JSON directly under the JSONB `roadmap_metadata` column on the `career_roadmaps` table, bypassing the need for unnecessary new tables.

### 13.2 Milestone Planner

Each milestone generated in the timeline strictly includes:
- `id`: A unique string identifier.
- `title`: Short label.
- `description`: Detailed description.
- `start_week` / `end_week`: Operational bounds.
- `estimated_hours`: Completion effort estimates.
- `prerequisite_skills`: Blockers or prior dependencies.
- `expected_outcome`: Tangible artifact or capability.
- `completion_status`: NOT_STARTED, IN_PROGRESS, or COMPLETED.

### 13.3 Weekly Planning & Monthly Plan

The system generates:
- **Weekly Goals**: Step-by-step topics and daily tasks list.
- **Monthly Milestones**: High-level focus milestones.
- **Quarterly Objectives**: Milestone targets for 3, 6, 12, 18, 24 months.

## 14. Career Analytics & Dashboard Engine

The Career Analytics & Dashboard Engine (Phase 11 Part 5) provides real-time progress calculations, metrics consolidation, and unified readiness evaluations across all user career roadmap states.

### 14.1 Key Capabilities & Statistical Engines

1. **Overall Portfolio Summary Metrics**: Aggregates average readiness scores, roadmap completions, completed milestones, and estimated vs. actual learning hours across all finished roadmaps.
2. **Roadmap Progress & Velocity calculations**: Calculates completed percentage, weekly velocity (milestones completed per week), delay analysis, and projection models, including weekly/monthly/quarterly/yearly progress intervals.
3. **Normalized Career Readiness Score**: Computes a unified `0-100` score using weighted parameters across the candidate profile:
   * **Resume ATS Match Score** (30% weight)
   * **AI Resume Review Overall Score** (25% weight)
   * **AI Resume Optimization Score** (15% weight)
   * **Mock Interview Score Average** (20% weight)
   * **GitHub Developer Index** (10% weight)
   If any of these metrics are missing, the system dynamically redistributes weight or falls back to skill gap progress indicators.
4. **Skill Difficulty & Domain Distribution**: Categorizes and displays skill densities, difficulty levels (BEGINNER, INTERMEDIATE, ADVANCED), and category distributions.
5. **Timeline Health Engine**: Scans active milestones to flag delays, classifying roadmap timeline health as `HEALTHY`, `AT_RISK`, or `BEHIND_SCHEDULE`.

### 14.2 API Endpoint Specifications

* **GET `/api/v1/ai/roadmap/analytics`**
  * *Summary*: Overall user roadmap statistics summary.
  * *Response*: Dictionary containing portfolio aggregates.
* **GET `/api/v1/ai/roadmap/{id}/analytics`**
  * *Summary*: Full cached roadmap dashboard metrics payload.
  * *Response*: `RoadmapAnalyticsResponse`.
* **GET `/api/v1/ai/roadmap/{id}/progress`**
  * *Summary*: Progression details, intervals, velocity, and delays.
  * *Response*: `ProgressResponse`.
* **GET `/api/v1/ai/roadmap/{id}/readiness`**
  * *Summary*: Normalized profile readiness indicators and action items.
  * *Response*: `CareerReadinessResponse`.
* **GET `/api/v1/ai/roadmap/{id}/skills`**
  * *Summary*: Detailed skill gaps, aggregates, and distributions.
  * *Response*: `SkillAnalyticsResponse`.

### 14.3 Cache Policy & Validation

Analytics calculations are dynamically cached inside `metadata` (`"analytics"`) of the roadmap record.
* **Invalidation Trigger**: Before returning cache, the service compares the database modified date `updated_at` (for the roadmap and all its milestone relationships) against the `generated_at` timestamp stored in cache.
* **Transaction Tolerance**: A 5-second buffer is used to prevent race-conditions or commit delay mismatches.

### 14.4 Privacy & Regulation Compliance

All analytics logging strictly excludes sensitive personal data, user profiles, or raw answers. The logs only trace operational data:
* `roadmap_id`
* `analytics_latency`
* `cache_hits` (1 or 0)
* `provider`
* `execution_time`
